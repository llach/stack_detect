The readme should explain how to run the component. It should also provide a component definition based on the following structure from D4.1: 

### Name of the Component

A brief, descriptive name for the component.

### Description

A short description of what the component does and its purpose.

### Functionality

A list of the key functions and features of the component, including input and output requirements.

### Technical Specifications

Details about the component’s technical requirements, such as:

- Programming language(s)

- Dependencies (e.g. libraries, frameworks)

- API/Interfaces 

- Operating system requirements 

- Hardware requirements (if any)

### Design

A description of the component’s design, including:

- Architecture 

- Data structures 

- Algorithms 

- Class and module structure (if applicable)

### Use Cases

Examples of how the component might be used in real-world scenarios and how it applies for SOFTENABLE use cases

### Testing

Details about how the component will be tested, including:

- Test cases 

- Test environment 

- Test data

### Limitations

Any known limitations or constraints of the component, such as performance limitations, compatibility issues, or other restrictions.
